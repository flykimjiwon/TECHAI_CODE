# 모델 한계 기록 (Model Limitations Log)

> 시연/운영 중 발견된 모델별 한계와 대응 방안. 발표 자료 참고용.

---

## GPT-OSS-120B (Novita.ai)

### 1. apply_patch 다중 hunk 실패
- **증상**: 2개 이상 hunk를 가진 패치 생성 시 50% 확률로 첫 hunk만 생성
- **원인**: 구조화된 출력(JSON + patch format) 동시 생성 능력 부족
- **대응**: auto-prefetch 시 apply_patch 도구 제거, file_write 강제
- **상위 모델 예상**: DeepSeek V3.2는 다중 hunk 안정적 생성 가능

### 2. 빈 응답 (Empty Response / Thinking-only)
- **증상**: completion 토큰 400~600개 생성하지만 content=0bytes, toolCalls=0
- **원인**: reasoning/thinking 토큰이 생성되나 API가 content로 전달하지 않음
- **대응**: 3회 재시도 + 점진적 컨텍스트 압축 (33% → 25% → 20%)
- **빈도**: 도구 6~8회 호출 후 발생 확률 높음

### 3. 시스템 프롬프트 과부하
- **증상**: 71KB 시스템 프롬프트 + 도구 결과 누적 → 빈 응답 급증
- **원인**: 128K 컨텍스트에서 프롬프트가 너무 크면 응답 생성 공간 부족
- **대응**: 81개 지식 문서 목록 제거, Knowledge Packs로 선택적 로드
- **수치**: 프롬프트 71KB → ~30KB로 줄인 후 성공률 대폭 향상

### 4. 도구 선택 비결정성
- **증상**: 동일 프롬프트에 file_write / apply_patch를 랜덤 선택
- **원인**: temperature > 0에서의 비결정적 도구 선택
- **대응**: prefetch 시 apply_patch/file_edit 도구 자체를 제거하여 강제
- **상위 모델 예상**: 지시를 더 잘 따르므로 도구 제거 불필요

### 5. 반복 패턴 루프
- **증상**: 같은 apply_patch 1-hunk → 읽기 → 같은 1-hunk 반복 (20회 tool limit)
- **원인**: 이전 시도가 왜 부분 성공했는지 인식하지 못하고 같은 전략 반복
- **대응**: 부분 성공 시 "file_write를 대신 사용하세요" 경고 메시지 추가
- **상위 모델 예상**: 실패 원인 인식 → 전략 전환 가능

### 6. 한국어 입력의 경로 파싱
- **증상**: "src/views/HomePage.tsx의" → 조사가 파일 경로에 붙어 인식 실패
- **원인**: 한국어 조사가 띄어쓰기 없이 경로에 결합
- **대응**: 파일 확장자 이후 문자를 모두 잘라내는 방식 (언어 무관)
- **참고**: 이건 모델 한계가 아니라 입력 전처리 문제이지만, 한국어 환경 특수성
