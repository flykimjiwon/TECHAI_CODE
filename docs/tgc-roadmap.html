4 멀티에이전트 아키텍처 설계

슈퍼택가이 모드에서 GPT-OSS-120B가 작업을 분석하고, Qwen3-Coder-30B 워커 3개에 분배 후, 결과를 통합 리뷰하는 파이프라인.

1. GPT-OSS-120B — 리드 (분석 & 계획)

사용자 요청 분석 → 작업을 3개 이하 서브태스크로 분해
각 서브태스크에 필요한 파일, 컨텍스트 정리

2. GPT-OSS-120B — 리뷰 (통합 & 검증)

워커 결과물 취합 → 충돌 해결 → 코드 리뷰
최종 결과 사용자에게 전달

동작 흐름 상세

단계모델동작토큰 예산
1. 분석 GPT-OSS-120B 사용자 요청 이해, 관련 파일 읽기, 서브태스크 분해 (JSON 출력) ~2000 tok
2. 분배 택가이코드 서브태스크 JSON 파싱 → Qwen3 워커에 각각 전달 0 (로컬)
3. 실행 Qwen3 x3 (병렬) 각자 파일 읽기 → 코드 작성/수정 → tool call 실행 ~3000 tok x3
4. 수집 택가이코드 3개 워커 결과물 수집, 변경 파일 목록 취합 0 (로컬)
5. 리뷰 GPT-OSS-120B 결과물 전체 리뷰, 충돌 해결, 품질 검증, 최종 답변 ~3000 tok

장점

- 120B 추론력으로 정확한 분석
- Qwen3 병렬로 3배 빠른 실행
- 120B 리뷰로 품질 보장
- Claude Code Team과 동일 패턴

고려사항

- 워커 간 파일 충돌 방지 필요
- 서브태스크 분해 품질이 핵심
- 단순 작업에는 오버헤드
- 워커 3개 동시 GPU 부하

구현 전략

- 단순 작업: 직접 120B 실행 (fallback)
- 복잡 작업: 자동 분해 → 워커 분배
- 판별 기준: 파일 2개+ 수정 시
- Go goroutine으로 병렬 호출

핵심: 모든 작업에 멀티에이전트를 쓰면 오버헤드. "파일 하나 수정해줘"는 120B 직접 처리. "이 프로젝트에 인증 시스템 추가해줘"처럼 복잡한 요청일 때만 자동으로 멀티에이전트 모드 진입. 판별은 GPT-OSS-120B가 첫 분석 단계에서 결정.

5 컨텍스트 최적화 전략

시스템 프롬프트에 모든 정보를 상시 주입하면 토큰 낭비. 하이브리드 방식 권장.

전략상시 토큰장점단점
A. 상시 전체 주입 ~500 tok 단순, 확실 매 요청마다 불필요한 IP/파일목록 포함
B. 최소 상시 + 도구 (권장) ~50 tok 토큰 최소, 필요시 shell_exec로 조회 AI가 도구 사용 판단 필요
C. 컨텍스트 파일 (Obsidian식) ~50 tok + 필요분 키워드 감지로 필요한 것만 로드 키워드 감지 로직 필요

권장: B+C 하이브리드

상시 주입 (~50 tok)

- OS/arch
- CWD (전체 경로)
- 현재 날짜/시간

이것만으로 기본 환경 파악 가능

키워드 감지 주입

- "IP" "네트워크" → IP 정보
- "파일" "목록" → 디렉토리 리스팅
- "깃" "브랜치" → git status
- "환경" "변수" → env 정보

나머지 → 도구 호출

- 디스크 용량 → shell_exec df
- 프로세스 목록 → shell_exec ps
- 네트워크 상태 → shell_exec ping
- 상세 시스템 정보 → shell_exec

싱글 바이너리 호환: Go 표준 라이브러리만으로 구현 가능. ~/.tgc/context/ 에 JSON/MD 파일 저장. 외부 의존성 제로. 빌드에 영향 없음.
