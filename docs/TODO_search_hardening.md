# 파일 서치 멈춤 원인 분석 & 강화 플랜

## 현재 증상
- 파일 500개+ 프로젝트에서 grep/glob 중 프로그램 멈춤
- 소스코드 수천 줄 파일에서 검색 시 응답 없음
- 간헐적으로 연결중 표시 없이 멈춤

## 의심 원인

### 1. Go regexp 성능 한계 (가장 유력)
- Go의 `regexp` 패키지는 NFA 기반 — 복잡한 패턴에서 O(n²) 가능
- 특히 `.*` + 백트래킹이 많은 패턴에서 느려짐
- 5MB 파일 × 복잡한 정규식 = 수 초~수십 초 블로킹
- **해결**: ripgrep 폴백 (이미 구현), 또는 정규식 타임아웃 추가

### 2. 대규모 디렉토리 WalkDir 블로킹
- `filepath.WalkDir`는 순차 탐색
- node_modules 같은 거대 디렉토리를 .gitignore가 못 잡으면 수만 파일 순회
- **해결**: 
  - 디렉토리별 타임아웃 (개별 파일 스캔 1초 제한)
  - 병렬 WalkDir (goroutine pool)
  - 파일 수 하드 리밋 (현재 5000 → OK)

### 3. 단일 파일 스캔 무한 루프
- `searchFile`이 큰 파일에서 모든 줄을 메모리에 로드 (`allLines`)
- 5MB 파일 = ~50,000줄 → 메모리 압박 + GC 지연
- **해결**: 스트리밍 방식으로 변경 (줄 단위 매칭, 전체 로드 않음)

### 4. LLM 도구 루프 타임아웃 없음
- AI가 grep → 결과 부족 → 다시 grep → 반복
- 20회 도구 루프 제한은 있지만 각 도구에 개별 타임아웃 없음
- **해결**: 개별 도구 실행에 10초 타임아웃 추가

### 5. 스트리밍 응답 중단
- LLM API에서 청크가 안 오면 무한 대기 (60초 후에야 감지)
- 네트워크 불안정한 사내망에서 빈번
- **해결**: 첫 청크 15초, 이후 30초 타임아웃 + 자동 재시도 (최대 3회)

## 강화 플랜 (우선순위순)

### Phase 1: 즉시 적용 (다음주)
1. **개별 파일 스캔 타임아웃** — searchFile에 context.WithTimeout(2초)
2. **스트리밍 방식 검색** — allLines 전체 로드 대신 줄 단위 매칭
3. **자동 재시도** — 15초 무응답 시 재연결 (최대 3회)
4. **병렬 WalkDir** — goroutine pool (worker 4개)

### Phase 2: 안정화
5. **정규식 복잡도 검사** — 위험한 패턴 감지 + 경고
6. **진행 표시** — "Searching... (247/1203 files)" 실시간 카운터
7. **취소 지원** — Esc로 진행 중인 검색 중단

### Phase 3: 고도화
8. **증분 인덱스** — 첫 스캔 후 변경 파일만 재스캔
9. **캐시** — 최근 검색 결과 캐싱 (동일 패턴 재검색 즉시 반환)
10. **ripgrep 스트리밍** — cmd.Output() 대신 StdoutPipe로 줄 단위 수신
