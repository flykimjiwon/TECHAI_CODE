0 JS/TS vs Go 생태계 — 전체 그림

언어 & 런타임

JS/TS 세계Go 세계
JavaScript (브라우저 태생)→Go (서버/시스템 태생)
TypeScript (JS + 타입)→Go는 처음부터 정적 타입
Node.js (JS를 서버에서 실행)→Go는 자체 런타임 내장 (별도 설치 불필요)
V8 엔진 (JIT 컴파일)→Go 컴파일러 (AOT → 네이티브 바이너리)

웹 프레임워크 여정

JS/TS 여정Go 대응
HTML/CSS/JS (순수 프론트)→Go는 프론트엔드 없음 (서버/CLI 전문)
React (컴포넌트 UI)→Bubble Tea (터미널 TUI) ← 택가이코드
Express.js (백엔드 첫 입문)→net/http (표준 라이브러리) 또는 Gin, Echo, Fiber
Next.js (풀스택 = React+API)→Go는 프론트/백 분리가 기본. 백엔드만 Go, 프론트는 Next.js
핵심 차이 JS는 "프론트 → 백엔드로 확장"하는 여정이지만, Go는 처음부터 서버/CLI/인프라 전문. 프론트엔드는 안 한다.

왜 Go를 쓰나?

JS/TS의 불편함Go에서 해결
node_modules 지옥의존성 최소, 표준 라이브러리 풍부
"works on my machine"단일 바이너리, 어디서든 실행
콜백/Promise 지옥goroutine으로 동시성 쉽게
런타임 타입 에러컴파일 타임에 다 잡힘
빌드 도구 설정 지옥go build 하나면 끝
서버 메모리 1GB+Go 서버 50MB로 동일 트래픽
Go가 약한 것 프론트엔드 (React/Next.js 압도적) · 빠른 프로토타이핑 (JS가 빠름) · 제네릭 (TS만큼 유연하지 않음) · 에러 핸들링 반복 (if err != nil 지옥)

3 함수 & 에러 핸들링

TypeScript (try/catch)
function divide(a, b) {
  if (b === 0)
    throw new Error("zero")
  return a / b
}
try {
  const r = divide(10, 0)
} catch(e) {
  console.error(e)
}
Go (에러를 리턴값으로!)
func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, fmt.Errorf("zero")
    }
    return a / b, nil
}

result, err := divide(10, 0)
if err != nil {
    log.Fatal(err)
}
Go의 철학: try/catch 없음! 에러는 항상 리턴값으로 전달. 매번 if err != nil로 체크. 번거롭지만 에러를 절대 무시할 수 없게 강제한다.

택가이코드 실제 예시 (config.go)

func Load() (Config, error) {
    cfg := DefaultConfig()
    data, err := os.ReadFile(ConfigPath())
    if err == nil {
        if err := yaml.Unmarshal(data, &cfg); err != nil {
            return cfg, fmt.Errorf("parse error: %w", err) // %w = 에러 래핑
        }
    }
    return cfg, nil
}

5 인터페이스 — 암시적 충족

TypeScript (명시적)
interface Animal {
  speak(): string
}
class Dog implements Animal {
  speak() { return "멍멍" }
}
Go (Duck Typing)
type Animal interface {
    Speak() string
}
type Dog struct{}
func (d Dog) Speak() string {
    return "멍멍"
}
// implements 없이 자동 충족!
Bubble Tea 인터페이스 (택가이코드 핵심) Init(), Update(), View() 3개 메서드만 구현하면 tea.Model 자동 충족. implements 키워드 없음.

6 고루틴 & 채널 — async/await 대신

TypeScriptGo
Promise→channel
await promise→<-channel (채널에서 값 꺼내기)
async function→go func() { }
Promise.all()→sync.WaitGroup

택가이코드 실제 패턴: LLM 스트리밍

func (c *Client) StreamChat(ctx context.Context, ...) <-chan StreamChunk {
    ch := make(chan StreamChunk)  // 채널 생성 (파이프)
    go func() {                   // 별도 goroutine에서 실행
        defer close(ch)             // 함수 끝날 때 채널 닫기
        for {
            resp, err := stream.Recv()
            if errors.Is(err, io.EOF) {
                ch <- StreamChunk{Done: true}
                return
            }
            ch <- StreamChunk{Content: delta}  // 실시간 전송
        }
    }()
    return ch
}

// 사용 측 (app.go)
chunk := <-m.streamCh  // ← 이게 await 같은 것

7 Context — 타임아웃 & 취소

// 30초 타임아웃으로 쉘 명령 실행 (tools/shell.go)
ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()  // 반드시 취소 (리소스 정리)

result, err := exec.CommandContext(ctx, "sh", "-c", command).Output()

// 스트리밍 취소 (사용자가 Ctrl+C 누를 때)
ctx, cancel := context.WithCancel(context.Background())
m.streamCancel = cancel     // 저장
m.streamCancel()            // 호출하면 goroutine 즉시 중단
비유 fetch의 AbortController 같은 것. 근데 Go에서는 모든 곳에서 사용한다.

8 defer — 정리 코드 보장

func processFile(path string) error {
    f, err := os.Open(path)
    if err != nil { return err }
    defer f.Close()  // ← 이 함수가 어떻게 끝나든 Close() 실행됨!
    data, err := io.ReadAll(f)
    // ...
}

// 택가이코드: 뮤텍스 잠금/해제
debugMu.Lock()
defer debugMu.Unlock()  // Lock → 작업 → 자동 Unlock
TypeScript의 finally를 자동화한 것. 파일 닫기, 뮤텍스 해제, 커넥션 반환 등 정리 코드를 "열 때" 바로 등록.

12 Bubble Tea — Go판 React

ReactBubble Tea
useState / useReducer→Model struct
dispatch(action)→Update(msg) → (Model, Cmd)
return <JSX />→View() → string
useEffect→Cmd (사이드 이펙트)
props→Msg (메시지)
// 1. 상태 (React의 state)
type Model struct {
    msgs      []ui.Message
    streaming bool
    activeTab int
}

// 2. 업데이트 (React의 reducer)
func (m Model) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
    switch msg := msg.(type) {
    case tea.KeyPressMsg:
        switch msg.String() {
        case "enter":    // 메시지 전송
        case "tab":      m.activeTab = (m.activeTab + 1) % 3
        case "ctrl+c":   return m, tea.Quit
        }
    case streamChunkMsg:  // 커스텀 메시지 (LLM 스트림)
        m.streamBuf += msg.content
    }
    return m, nil
}

// 3. 렌더링 (React의 render)
func (m Model) View() tea.View { /* 문자열로 UI 구성 */ }

2. 미사용 변수/import = 컴파일 에러 import "fmt" 해놓고 안 쓰면 에러!   _ = x로 의도적 무시.

슬라이스 (배열)

arr := []string{"a", "b", "c"}   // 생성
arr = append(arr, "d")             // push

for i, item := range arr {          // forEach with index

map (객체)

m := map[string]int{}              // Record<string, number>
m["key"] = 42                      // 값 설정
val, ok := m["key"]                // 값 읽기 (ok = 존재 여부)
delete(m, "key")                   // 삭제

15 전환 체크리스트

  • 에러 = 리턴값 — try/catch 없음, if err != nil 반복
  • 대문자 = public — export 키워드 없음
  • goroutine + channel = async/await
  • context = 취소/타임아웃 전파
  • defer = finally 자동화
  • interface = 암시적 충족 (implements 없음)
  • 단일 바이너리 배포 (런타임 불필요)
  • go build로 크로스 컴파일 (GOOS/GOARCH)
  • 미사용 = 컴파일 에러 (깔끔한 코드 강제)
