# 아키텍처 개선 방향 — 룰베이스 + 마이크로 에이전트

> 시스템 프롬프트/컨텍스트에 의존하지 않고, 프로그램 자체의 로직으로 품질을 보장한다.

---

## 현재 문제

시스템 프롬프트에 규칙을 넣으면:
- 약한 모델은 무시함 (50% 확률)
- 강한 모델에게는 불필요한 토큰 낭비
- 모델이 바뀌면 규칙도 재조정 필요

## 개선 원칙

```
프롬프트 규칙 (불안정) → 코드 로직 (확정적)
```

---

## 1. 룰베이스 하드코딩

모델에게 "하지 마"라고 말하는 대신, **프로그램이 차단**.

| 현재 (프롬프트) | 개선 (코드) |
|---------------|-----------|
| "NEVER force push" | `shell.go`에서 `git push --force` 패턴 차단 |
| "NEVER write .env" | `file.go`에서 `.env` 경로 쓰기 거부 |
| "Read before edit" | `file_edit` 호출 시 자동으로 file_read 선행 |
| "No recursive on node_modules" | `list_files`에서 skipDirs 하드코딩 (이미 구현) |

## 2. RAG 기반 맥락 주입 (현재 Knowledge Packs 확장)

현재: 키워드 매칭 → 문서 주입
개선: **질문 의도 분류 → 관련 컨텍스트만 정밀 주입**

```
사용자: "React Hook Form으로 폼 만들어줘"
    ↓
[의도 분류] 코드 생성 + React + Form
    ↓
[RAG] react-hook-form.md에서 관련 섹션만 추출 (전체 문서 X)
    ↓
[주입] 500토큰만 시스템 프롬프트에 추가 (전체 2000토큰 X)
```

## 3. 마이크로 에이전트 분산

하나의 거대 모델이 모든 걸 하는 대신, **세부 도구마다 전문 에이전트**.

```
[메인 에이전트] GPT-OSS-120B — 의도 파악 + 작업 분배
    ├── [검색 에이전트] Qwen3-Coder-30B — grep/glob 실행 + 결과 요약
    ├── [편집 에이전트] Qwen3-Coder-30B — file_write 전용 (코드 생성 특화)
    ├── [검증 에이전트] Qwen3-Coder-30B — 빌드/린트 실행 + 에러 분석
    └── [리뷰 에이전트] Qwen3-Coder-30B — 수정 결과 검토
```

장점:
- 각 에이전트는 짧은 프롬프트 + 단일 역할 → **3B active로도 충분**
- 메인 에이전트는 분배만 → 빠름
- 에이전트별 독립 실패 → 전체 실패 방지
- 병렬 실행 가능

### 예시 흐름

```
사용자: "HomePage에 거래내역 추가해줘"

[메인] 의도 파악: 코드 수정
  → [검색] grep "HomePage" → 파일 위치 확인
  → [검색] grep "transactions" → 데이터 위치 확인
  → [편집] HomePage.tsx 전체 재작성 (file_write)
  → [검증] npm run build → 성공 확인
  → [메인] 사용자에게 결과 보고
```

## 4. 하드코딩 자동 검증 루프

현재: 수정 → 끝
개선: **수정 → 빌드 → 에러 시 자동 수정 → 재빌드**

```go
// 의사 코드
for i := 0; i < 3; i++ {
    agent.Edit(file)
    result := shell.Exec("npm run build")
    if result.Success {
        break
    }
    agent.FixError(result.Error) // 에러를 보고 자동 수정
}
```

이건 프롬프트가 아니라 **Go 코드**로 구현. 모델 능력과 무관하게 동작.

---

## 우선순위

1. **룰베이스 하드코딩** — 즉시 가능, 프롬프트 규칙을 코드로 이전
2. **자동 검증 루프** — build/lint 자동 실행, 에러 자동 수정
3. **RAG 정밀 주입** — 키워드 → 의도 분류 → 섹션 추출
4. **마이크로 에이전트** — 장기 과제, 멀티 에이전트 재설계 시 적용
